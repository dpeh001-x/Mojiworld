Mobile playtest updates

Mobile-only changes shipped on branch mobile-ui-pass. Keeps the main CHANGELOG.html focused on gameplay releases. Every entry here corresponds to an atomic commit on the mobile branch.

Branch: mobile-ui-pass · Preview: play now

Mobile playtest batch — F works again · Block hidden · desktop-keys strip gone bug

Three fixes from a live mobile run

User report: "F button doesn't work anymore. Block button still visible. The controls like Enter, N don't exist on phone." All three addressed together:

  1. F button dynamic dispatch (restored). The reverted dynamic-F feature is back as a JS-only implementation (no CSS visual switcharoo this time — that's what risked the layout regression before). On tap, F picks its dispatch key at press time: nearby chest / NPC → 'f' (tryInteract fires); no interact target + Q unlocked at Lv 25+ → 'f' (cast Q); no interact + Q locked → 'a' (block / parry). The keyup dispatches the same key keydown used, so game.keys can't get stuck. Implementation: added data-dynamic="f-block" attribute on the F button, made _mkeyBind skip buttons with data-dynamic, and wired a dedicated _wireDynamicFButton.
  2. Block (🛡) button hidden on mobile. Dynamic F already dispatches 'a' when no interact is in range, so the separate floating Block circle was redundant. CSS .mobile-deck .mc-block { display: none !important; } inside the mobile media query. Caveat: at Lv 25+ with Q unlocked, F casts Q instead of blocking — face-cluster block is lost. Desktop 'a' key still works. The earlier landscape-Block override (B1) is now effectively dead code but left in place as insurance if Block is ever un-hidden.
  3. #controls help strip hidden on mobile. The bottom-left strip listing desktop keys (? Controls · Enter Chat · N Talk/Codex · B Bag · R/T Pots · M Mute) referred to keys that don't exist on touch devices. CSS #controls { display: none !important; } inside the mobile media query.

Audit P1.2 — skill-point toast on level-up feat

Mobile players no longer miss the +1 SP on level-up

gainExp() plays the levelup fanfare + spawns particles + flashes a "LEVEL UP!" element, but mobile players have no obvious cue that a new Skill Point was added or where to spend it (Skills panel is hidden behind the 📖 button in the pots row).

Fix: gated IIFE polls player.skillPoints every 500ms. On increase, fires showToast('+N Skill Point — tap 📖 to spend', 'rare'). Reuses the existing toast pipeline. Zero gameplay code touched — only reads player.skillPoints.

Audit P1.10 — tap the death overlay to skip the 1.6s wait feat

Mobile players no longer stare at a frozen screen on death

The game loop sets game.dying = 1600 on death and ticks it down; when it reaches 0, respawnAtTown() fires automatically. Desktop players can keep moving / pressing keys so it feels like they're doing something; mobile players stare at a frozen "DEATH" screen for 1.6 seconds with no feedback that anything is happening.

Fix: enabled pointer-events: auto on #death-overlay.on inside the mobile media query + added a gated IIFE that attaches pointerdown to the overlay. On tap, if game.dying > 0, set game.dying = 0. The existing game loop picks it up next frame and runs respawnAtTown() immediately. Zero gameplay code touched — the listener only reads/writes game.dying, which the game loop already drives.

Correctness nits: KeyboardEvent repeat + MutationObserver refs polish

Defensive fixes caught in an audit sweep

Synthetic keyboard events: _mkeyDispatch was creating KeyboardEvents without the repeat field; the global keydown handler reads e.repeat to skip fresh-press logic. undefined is falsy so it works today, but any future handler comparing e.repeat === false would silently break on synthetic events. Added explicit repeat: false to the init dict.

MutationObservers: the _forcedModals watcher was calling new MutationObserver(...).observe(...) without saving the returned object. The observers only stay alive because observe() pins them via the DOM — any later audit would assume they're leaked. Now stored in window._forcedModalObservers so the reference is explicit.

Rotation refit thrash — rAF debounce polish

One rotation was triggering up to 10 refits

_fitGameToViewport was bound to resize + orientationchange + screen.orientation.change + visualViewport.resize + a 6-step setTimeout tail (0 / 120 / 350 / 800 / 1400 / 2000ms). A single rotation could fire 10+ refits — visible as layout jitter.

Fix: new _fitSoon() wrapper coalesces back-to-back calls inside one animation frame into a single real fit. Every event path now goes through _fitSoon. The 6-step timeout schedule stays (iOS needs the tail for URL-bar-settle), but each step goes through the rAF gate — so at most one fit per frame.

Narrow-phone CSS specificity sweep bug

≤380px portrait phones were getting the wrong button sizes

The @media (max-width: 380px) block at maple_game.html:385 shrank buttons for narrow phones but several rules (.mc-actions grid template + size, .mc-dpad .mc-up/down/left/right positions, .mc-dpad::before inset) were missing !important. The portrait rules above them (line 322–364) do use !important, so on a narrow-portrait device both media queries match and portrait wins the specificity tie — leaving narrow phones with 160×160 face cluster instead of 172×172 and misaligned d-pad arrows.

Fix: added !important to every property in the narrow-phone block (~8 rules). Now narrow-portrait devices actually receive the designed sizing.

Landscape Block override bug

Block was sitting directly on top of the d-pad in landscape

Default mobile Block position is left: 54px, bottom: 190px — designed for portrait where the d-pad top edge is at ~180px from viewport bottom. Landscape viewports are shorter; d-pad at bottom: 18, height ~168 puts its top edge at ~186px. Block's bottom edge at 190 + 54px height overlapped the d-pad's top row by ~50px.

Fix: added a body.mc-landscape .mobile-deck .mc-block override that pushes Block to left: 186px (right of the d-pad) at bottom: 60px (mid-d-pad height). Block stays left-hand reachable without stealing the d-pad footprint. Portrait unaffected.

F button doubles as Block — dynamic dispatch feat

Context-aware F: interact nearby → F, else → Block

User suggestion: "Since F is only to enter portals or open chest, can it be dynamic. When there is no portal it becomes the block button". Implemented. F now dispatches one of two keys depending on live game state at press time:

  • Nearby chest / portal / NPC: dispatches F — runs tryInteract() and casts the Q skill if unlocked.
  • No interact available, Q skill unlocked (Lv 25+): dispatches F — cast Q. Kept reachable late-game so Q doesn't disappear on mobile.
  • No interact available, Q skill locked: dispatches A — block / parry. Hold to block, tap-time parry window applies.

Visual updates every 200ms: letter F with amber border in interact/Q mode; shield 🛡 with blue border in block mode. Skill-lock grey-out is skipped on F (same as before — F is never "dead" with this change). The separate floating Block button is hidden now since F covers its role pre-Lv 25.

Late-game caveat: at Lv 25+ with Q unlocked, F always dispatches 'f' (no block fallback) — Q casting stays accessible but block needs the desktop A key. If this is annoying in practice we can re-add the Block button.

Sizes: 4 × 48px + 3 × 8px gap = 216px wide in portrait (vs 160 before). Landscape override removed (no longer needed).

Portrait shrink + grey-out accuracy + F exemption polish

Follow-up on the label-sync commit

Three related fixes after live playtest feedback:

  1. Portrait was too crowded. On narrow phones the face cluster (184px) + d-pad (156px) + pots row ate too much of the viewport. Shrunk in portrait: cluster 184→160px, cells 54→48, gap 11→8; d-pad 156→140px, arrows 46→42; pots 44→40px. Landscape unchanged.

Face-cluster label sync + lock state polish

Mobile buttons now line up 1-to-1 with the skill bar

Two issues fixed together:

  1. Labels didn't match keys. The face cluster showed W A S D X E Q C (gamepad-style visual labels) but the actual keys each button fired were d s c f g z x s v. Players couldn't tell which face button corresponded to which skill-bar icon because the letters didn't line up. Relabeled so every face button shows its own key:
  [D] [↑] [V]       old: [W] [↑] [C]
  [S] [Z] [X]            [A] [D] [S]
  [G] [C] [F]            [X] [E] [Q]

Now a player seeing an icon on skill-bar slot F can look straight at face-button F to cast it. The skill bar at the bottom (A Z X S C D F V G) and the face cluster share the same letters.

  1. Locked skills weren't visually distinct. Skills that haven't unlocked yet (no match in the player's current kit — e.g. F needs the 1st job advancement at Lv 25) were fully visible on the face cluster, giving no hint that tapping them does nothing. New .mc-btn.mc-locked class toggled from a 500ms poll that mirrors _sbSlots[slot].skill — the same predicate the desktop skill bar uses for its own grey-out. Locked buttons dim to opacity: 0.32 + grayscale(1). Basic attack (Z) and Jump are never marked locked — they're always castable.

Fix: NPC dialog Leave/option buttons untappable on mobile bug

Same stacking-context trap the advancement modal had

When an NPC dialog opened (e.g. "Buy Potions" · "Leave" on a shopkeeper) on mobile, none of the option buttons responded to taps. Root cause is the same one the class-select + advancement modals already had: .game-wrapper has a transform: scale() that creates a stacking context, trapping every modal inside it below the #mobile-deck (z-index: 70, outside the wrapper). The deck's action buttons eat the taps even though the deck itself is pointer-events: none.

Fix: extended the existing _forcedModals watcher to include #dialog. Opening the NPC dialog now toggles body.forced-modal, which the existing CSS rule uses to hide the mobile deck + in-wrapper ctrl while the dialog is open. Closing the dialog restores them. Handles the style.display difference too — modals use 'flex', #dialog uses 'block' — via a { id, open } pair list.

Fix: orientation bounce leaves the game at the old scale bug

Landscape → portrait → landscape stuck in a portrait-sized box

After rotating the phone twice (landscape → portrait → landscape), the game canvas stayed at the small portrait scale and didn't refill the viewport. _fitGameToViewport listens for orientationchange + resize, but on iOS Safari the final settle (after the URL bar hides/shows) can arrive 1–2s after the orientation event fires — past the existing 800ms retry tail.

Fix: three additions:

  • Extended the retry schedule to 2000ms (added 1400 + 2000 beats) so slow iOS settles are caught.
  • Wired screen.orientation.addEventListener('change', …) — the modern replacement for orientationchange, fires after the browser finishes applying the change.
  • Wired visualViewport.addEventListener('resize', …) — the most reliable signal that the usable viewport actually changed (catches URL-bar show/hide on iOS).

Each is wrapped in a feature-detect + try/catch so older browsers without the API silently skip it.

P1.7 — Haptic vibrate on legendary drops feat

Physical cue for a physical rarity

When a legendary item drops, fire a [40, 30, 70] vibrate pattern (buzz-pause-buzz) via the Vibration API. The small physical response doubles the perceived weight of a rare pull — variable-ratio reward with a tangible cue.

Implementation: IIFE wraps the existing emitRarityBurst(x, y, rarity) at load via a runtime function-replacement (original source untouched, no gameplay-code edit). The wrapper calls through then fires navigator.vibrate only on rarity === 'legendary'. iOS Safari returns false from navigator.vibrate unconditionally, so this is a no-op there without any Safari-specific fallback path.

Audit lever: P1.7 (Variable-ratio +2, medium ROI · size S).

P1.9 — Next-advancement chip in HUD feat

Always see what level you're racing toward

Between Lv 1 → 25 and again 25 → 50, the player is grinding without any forward-looking cue about the next big unlock. Added a small amber chip under the HP/MP/EXP bars:

  • No job + Lv < 25 → → Job in N
  • Job only + Lv < 50 → → Master in N
  • Past threshold but not advanced → ✦ Press K to advance
  • Master already claimed → hidden

Implementation: IIFE injects #adv-chip as a child of #stats on DOMContentLoaded; 500ms interval refreshes the text. Uses JOB_ADVANCE_LEVEL / MASTER_ADVANCE_LEVEL constants (defined in gameplay code as read-only references). Zero gameplay mutation.

Audit lever: P1.9 (Progress overview +3, high ROI · size S).

P1.8 — Streak-at-risk warning on load feat

Passive streaks → loss-aversion cue

If the player has a streak of 2+ that isn't already claimed today AND less than 2 hours remain before UTC midnight, a toast fires on load:

⚠ Xh left to keep your N-day streak

...and the #daily-banner pulses for 3 seconds (amber glow). Loss aversion is ~2× the behavioural pull of gain — "don't lose your 6-day streak" hits harder than "log in for a bonus".

Implementation: IIFE polls up to 10× at 500ms waiting for game.dailyState to populate (save load is async), evaluates the three conditions (streak>=2, !claimed, ms<2h) and fires the toast via the existing showToast + adds a CSS .daily-pulse class. Both CSS and JS live inside the mobile zone.

Audit lever: P1.8 (Loss +3 · Re-engagement +2, high ROI · size S).

P1.5 — Daily-banner resets-in countdown feat

"Resets in 4h 12m" / "Resets in 0m 34s"

Previously the banner said what the challenge was and your progress, but not when it rolls over. Now a new line inside the banner ticks down to the next UTC midnight — the same boundary dailyIndex() uses for the rollover. Last hour switches to m:ss format for urgency.

Audit lever: P1.5 (FOMO +3 · Loss +2, high ROI · size S).

P1.4 — XP-bar near-level pulse feat

The bar draws the eye in the last 10% of a level

When player.exp / player.expToNext > 0.9, the #exp-bar gains a subtle filter: brightness(1) → 1.35 loop (1.1s ease). Drops off below 0.85 (hysteresis so it doesn't flicker at the threshold). Mobile-only — the desktop HUD is dense enough without the animation.

Implementation: CSS keyframe + IIFE polling player.exp / player.expToNext every 400ms, toggling the xp-near class on #exp-bar. Zero gameplay code touched.

Audit lever: P1.4 (Near-miss +3, high ROI · size S).

P1.1 — Tab-title badge on level-up feat

Backgrounded tab draws the eye when you level up

Mobile players browse freely across tabs — the only way to know something happened in LevelX while you were elsewhere is the tab icon. Added a badge: when the tab is hidden and player.level increases, the title flips to ⭐ Lv N — LevelX until the tab is focused again, at which point it restores to the baseline.

Implementation: self-contained IIFE that listens to visibilitychange. Goes document.hidden ➞ start a 500ms poll of player.level vs. the level recorded on hide. P1.3 — Rotate-nag dismissal persists across reloads feat

Stop re-prompting players who already said "play in portrait anyway"

The "Rotate your phone" overlay used to re-appear on every page reload because the dismissal only lived in document.body.dataset.nagDismissed (session-only). Players who prefer portrait were nagged every single time they opened the tab.

Fix: on dismissal, also write localStorage._mobile_nagDismissed = "1". On page load, read that back into dataset.nagDismissed before the first _fitGameToViewport call, so the nag never flashes. Both reads and writes are wrapped in try/catch to survive Safari private browsing (which throws SecurityError on localStorage access).

Clearing the dismissal is as simple as running localStorage.removeItem('_mobile_nagDismissed') in the console. From the audit: P1.3 (quit-point removal, high ROI · size S).

Fix: attack buttons unresponsive after smart d-pad landing bug

Removed setPointerCapture + reverted jump layout

After the smart d-pad + jump-promotion shipped, face-cluster attack buttons (basic D, skill S/A/E/W/Q, ults C/X) stopped responding to taps. Two regressions combined:

  1. iOS Safari setPointerCapture quirk. The smart-dpad grabbed pointer capture on the first touch; Safari then silently dropped simultaneous pointer events on other elements while capture was held. The d-pad still worked because its own pointer drove it, but every button in the right-hand cluster became inert for the duration of any d-pad press.
  2. Oversized jump A-button overflow. The 72px jump in a 54–58px grid cell with z-index:2 created a stacking context that, combined with the capture bug, made debugging the first issue harder.

Fix:

  • Replaced pad.setPointerCapture() with window-scoped pointermove / pointerup listeners that are attached on pointerdown and removed on release. Same UX — thumb can drag outside the pad and still hold the direction — but other touches now fire their own events independently.
  • Added a guard so a second finger landing on the pad while one is already active is ignored (no interleaved pointerIds).
  • Reverted jump back to top-center of the face cluster at its original 58px size per user request. Skill Q returns to its original bottom-right slot.

Smart floating d-pad feat

Drag-to-reverse without lifting your thumb

The movement d-pad used to be 4 separate buttons. Quick left↔right reversals required lifting the thumb, crossing the gap, and pressing again — missed inputs were common during fast platforming.

Fix: the .mc-dpad container is now a single continuous touch target. Touch anywhere inside; the thumb's angle from center resolves to one of 4 cardinal directions via Math.abs(dx) > Math.abs(dy). Drag mid-press to reverse without lifting. The existing 4 arrow buttons remain visible as direction hints and mirror the :active press state via a new .mc-active class toggled by the smart-dpad JS.

Details:

  • 14px dead-zone at center — a thumb resting exactly on center fires no direction.
  • setPointerCapture tracks the original pointer ID, so the thumb can slide outside the pad while still holding the direction.
  • Dispatches the same synthetic KeyboardEvents as before via _mkeyDispatch — no parallel code paths, no gameplay-code edits.
  • Individual d-pad arrow buttons now pointer-events: none; the parent container handles all input, so drags between arrows don't lose tracking.
  • Lives entirely inside mobile safe zones: CSS 144–184, JS _wireSmartDpad() at 3700–3755.

Face-button cluster (action cluster) still uses per-button pointer-events: auto — that's a discrete-action region, not a directional one.

Class & advancement taps unblocked bug

Mobile control deck was eating taps on the forced modals

At game start the class-select modal pops up, and at Lv 25 / 50 the job / master advancement modals do the same. On mobile, taps on the class / job / master cards were being swallowed before they reached the modal — you'd tap a card and nothing would happen.

Root cause: .modal-overlay sits inside .game-wrapper, which uses transform: scale(). That transform creates a new stacking context, so the modal's z-index: 50 is trapped inside the wrapper. #mobile-deck is a sibling of the wrapper at z-index: 70, and its d-pad / face-button / skill-grid buttons have pointer-events: auto — so wherever a control button overlapped a card, the card couldn't be tapped.

Fix:

  • New CSS rule: body.forced-modal .mobile-deck, body.forced-modal #mobile-ctrl, body.forced-modal #mobile-ctrl-toggle { display: none !important; }. Lives inside the MOBILE CONTROLS (v5) safe zone (lines 65–334).
  • New JS in _initMobileControls: a tiny MutationObserver watches style on #class-select-modal and #advancement-modal. When either is display: flex, it sets body.forced-modal; otherwise it removes it. The game is already paused while these modals are up, so hiding the deck has no gameplay cost.
  • Bonus cleanup: the MOBILE / TOUCH CONTROLS JS safe-zone range in the guard hook + CLAUDE.md was previously mis-set — it started at line 3317, which was actually deep inside getAtk() (gameplay code). Corrected to 3612–3797, the real boundaries of the mobile JS block.

This also fixes the Lv 25 / Lv 50 advancement modals, which use the same .class-card styling and were suffering the same tap-eating behaviour.

  • All rules scoped to @media (pointer: coarse), (max-width: 900px), inside the existing Mobile Controls (v5) CSS block — so desktop is untouched.

Verified via the mobile-zone guard hook: every change lands in line range 65–322 (the MOBILE CONTROLS (v5) safe zone) so there is no conflict risk with the collaborator's gameplay work on main.

Hardening pass before merging to main
