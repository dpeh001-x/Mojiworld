Fix: NPC dialog Leave/option buttons untappable on mobile bug

Same stacking-context trap the advancement modal had

When an NPC dialog opened (e.g. "Buy Potions" В· "Leave" on a shopkeeper) on mobile, none of the option buttons responded to taps. Root cause is the same one the class-select + advancement modals already had: .game-wrapper has a transform: scale() that creates a stacking context, trapping every modal inside it below the #mobile-deck (z-index: 70, outside the wrapper). The deck's action buttons eat the taps even though the deck itself is pointer-events: none.

Fix: extended the existing _forcedModals watcher to include #dialog. Handles the style.display difference too вҖ” modals use 'flex', #dialog uses 'block' вҖ” via a { id, open } pair list.

Fix: orientation bounce leaves the game at the old scale bug

Landscape вҶ’ portrait вҶ’ landscape stuck in a portrait-sized box

After rotating the phone twice (landscape вҶ’ portrait вҶ’ landscape), the game canvas stayed at the small portrait scale and didn't refill the viewport. _fitGameToViewport listens for orientationchange + resize, but on iOS Safari the final settle (after the URL bar hides/shows) can arrive 1вҖ“2s after the orientation event fires вҖ” past the existing 800ms retry tail.

Fix: three additions:

  • Extended the retry schedule to 2000ms (added 1400 + 2000 beats) so slow iOS settles are caught.
  • Wired screen.orientation.addEventListener('change', вҖҰ) вҖ” the modern replacement for orientationchange, fires after the browser finishes applying the change.
  • Wired visualViewport.addEventListener('resize', вҖҰ) вҖ” the most reliable signal that the usable viewport actually changed (catches URL-bar show/hide on iOS).

Each is wrapped in a feature-detect + try/catch so older browsers without the API silently skip it.

P1.7 вҖ” Haptic vibrate on legendary drops feat

Physical cue for a physical rarity

When a legendary item drops, fire a [40, 30, 70] vibrate pattern (buzz-pause-buzz) via the Vibration API. The small physical response doubles the perceived weight of a rare pull вҖ” variable-ratio reward with a tangible cue.

Implementation: IIFE wraps the existing emitRarityBurst(x, y, rarity) at load via a runtime function-replacement (original source untouched, no gameplay-code edit). The wrapper calls through then fires navigator.vibrate only on rarity === 'legendary'. iOS Safari returns false from navigator.vibrate unconditionally, so this is a no-op there without any Safari-specific fallback path.

Audit lever: P1.7 (Variable-ratio +2, medium ROI В· size S).

P1.8 вҖ” Streak-at-risk warning on load feat

Passive streaks вҶ’ loss-aversion cue

If the player has a streak of 2+ that isn't already claimed today AND less than 2 hours remain before UTC midnight, a toast fires on load:

P1.3 вҖ” Rotate-nag dismissal persists across reloads feat

Stop re-prompting players who already said "play in portrait anyway"

The "Rotate your phone" overlay used to re-appear on every page reload because the dismissal only lived in document.body.dataset.nagDismissed (session-only). Players who prefer portrait were nagged every single time they opened the tab.

Fix: on dismissal, also write localStorage._mobile_nagDismissed = "1". On page load, read that back into dataset.nagDismissed before the first _fitGameToViewport call, so the nag never flashes. Both reads and writes are wrapped in try/catch to survive Safari private browsing (which throws SecurityError on localStorage access).

Clearing the dismissal is as simple as running localStorage.removeItem('_mobile_nagDismissed') in the console. From the audit: P1.3 (quit-point removal, high ROI В· size S).

Fix: attack buttons unresponsive after smart d-pad landing bug

Removed setPointerCapture + reverted jump layout

After the smart d-pad + jump-promotion shipped, face-cluster attack buttons (basic D, skill S/A/E/W/Q, ults C/X) stopped responding to taps. Two regressions combined:

  1. iOS Safari setPointerCapture quirk. The smart-dpad grabbed pointer capture on the first touch; Safari then silently dropped simultaneous pointer events on other elements while capture was held. The d-pad still worked because its own pointer drove it, but every button in the right-hand cluster became inert for the duration of any d-pad press.
  2. Oversized jump A-button overflow. The 72px jump in a 54вҖ“58px grid cell with z-index:2 created a stacking context that, combined with the capture bug, made debugging the first issue harder.

Fix:

  • Replaced pad.setPointerCapture() with window-scoped pointermove / pointerup listeners that are attached on pointerdown and removed on release. Same UX вҖ” thumb can drag outside the pad and still hold the direction вҖ” but other touches now fire their own events independently.
  • Added a guard so a second finger landing on the pad while one is already active is ignored (no interleaved pointerIds).
  • Reverted jump back to top-center of the face cluster at its original 58px size per user request. Skill Q returns to its original bottom-right slot.

Smart floating d-pad feat

Drag-to-reverse without lifting your thumb

The movement d-pad used to be 4 separate buttons. Quick leftвҶ”right reversals required lifting the thumb, crossing the gap, and pressing again вҖ” missed inputs were common during fast platforming.

Fix: the .mc-dpad container is now a single continuous touch target. Touch anywhere inside; the thumb's angle from center resolves to one of 4 cardinal directions via Math.abs(dx) > Math.abs(dy). Drag mid-press to reverse without lifting. The existing 4 arrow buttons remain visible as direction hints and mirror the :active press state via a new .mc-active class toggled by the smart-dpad JS.

Details:

  • 14px dead-zone at center вҖ” a thumb resting exactly on center fires no direction.
  • setPointerCapture tracks the original pointer ID, so the thumb can slide outside the pad while still holding the direction.
  • Dispatches the same synthetic KeyboardEvents as before via _mkeyDispatch вҖ” no parallel code paths, no gameplay-code edits.
  • Individual d-pad arrow buttons now pointer-events: none; the parent container handles all input, so drags between arrows don't lose tracking.
  • Lives entirely inside mobile safe zones: CSS 144вҖ“184, JS _wireSmartDpad() at 3700вҖ“3755.

Face-button cluster (action cluster) still uses per-button pointer-events: auto вҖ” that's a discrete-action region, not a directional one.

Class & advancement taps unblocked bug

Mobile control deck was eating taps on the forced modals

At game start the class-select modal pops up, and at Lv 25 / 50 the job / master advancement modals do the same. On mobile, taps on the class / job / master cards were being swallowed before they reached the modal вҖ” you'd tap a card and nothing would happen.

Root cause: .modal-overlay sits inside .game-wrapper, which uses transform: scale(). That transform creates a new stacking context, so the modal's z-index: 50 is trapped inside the wrapper. #mobile-deck is a sibling of the wrapper at z-index: 70, and its d-pad / face-button / skill-grid buttons have pointer-events: auto вҖ” so wherever a control button overlapped a card, the card couldn't be tapped.

Fix:

  • New CSS rule: body.forced-modal .mobile-deck, body.forced-modal #mobile-ctrl, body.forced-modal #mobile-ctrl-toggle { display: none !important; }. Lives inside the MOBILE CONTROLS (v5) safe zone (lines 65вҖ“334).
  • New JS in _initMobileControls: a tiny MutationObserver watches style on #class-select-modal and #advancement-modal. When either is display: flex, it sets body.forced-modal; otherwise it removes it. The game is already paused while these modals are up, so hiding the deck has no gameplay cost.
  • Bonus cleanup: the MOBILE / TOUCH CONTROLS JS safe-zone range in the guard hook + CLAUDE.md was previously mis-set вҖ” it started at line 3317, which was actually deep inside getAtk() (gameplay code). Corrected to 3612вҖ“3797, the real boundaries of the mobile JS block.

This also fixes the Lv 25 / Lv 50 advancement modals, which use the same .class-card styling and were suffering the same tap-eating behaviour.

Verified via the mobile-zone guard hook: every change lands in line range 65вҖ“322 (the MOBILE CONTROLS (v5) safe zone) so there is no conflict risk with the collaborator's gameplay work on main.
